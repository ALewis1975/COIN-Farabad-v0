# CIVSUB Incident/Lead Permutation Matrix (Civilian-Involving)

## Purpose

Capture the incident/lead permutations that involve civilians, then classify whether each permutation should be integrated into CIVSUB and how.

## Current contracts in code (source of truth)

- Runtime incident types currently generated by the incident selector are: `LOGISTICS`, `ESCORT`, `IED`, `RAID`, `DEFEND`, `QRF`, `PATROL`, `RECON`, `CIVIL`, `CHECKPOINT`. These come from the incident weighting switch in `fn_incidentCreate.sqf`. 
- Follow-on lead generation on incident close is handled in `fn_leadGenerateFromIncident.sqf` and is keyed by `(result, incidentType)` where `result ∈ {SUCCEEDED, FAILED}`.
- CIVSUB currently captures an incident-window baseline at incident acceptance (`activeIncidentCivsubDistrictId`, `activeIncidentCivsubStart`) and appends a CIVSUB annex during SITREP ingest.
- CIVSUB interaction events (ID checks, aid, detention, questioning) emit district deltas and counters.
- CIVSUB can include `lead_emit` inside CIVSUB bundles (ambient lead scheduler + crime DB hit), but there is currently no bridge that materializes those `lead_emit` envelopes into the core `leadPool` via `ARC_fnc_leadCreate`.

## Decision legend

- **YES (Now):** Integrate in CIVSUB immediately.
- **YES (Guarded):** Integrate, but with explicit guardrails/caps.
- **LATER:** Defer until dedicated-server validation and/or additional policy decisions.
- **NO:** Keep out of CIVSUB for now.

## Required integration work (ordered)

1. **Lead bridge (highest value):**
   - Implement a server-side consumer for CIVSUB bundles where `lead_emit.emit == true`.
   - Normalize CIVSUB lead types to core incident types accepted by `ARC_fnc_leadCreate`/incident generator.
   - Write back created lead IDs for traceability.

2. **Incident district binding hardening:**
   - Ensure every accepted incident that touches civilians has `activeIncidentCivsubDistrictId` set deterministically.
   - Preserve a consistent start snapshot key shape (`activeIncidentCivsubStart*`) and use it in SITREP annex delta math.

3. **Annex -> policy feedback:**
   - Use annex deltas/counters as explicit gating hints for TOC closeout recommendations (not direct auto-close blockers).

4. **Guardrails:**
   - Keep server as single writer for CIVSUB district and shared state.
   - Cap CIVSUB-originated lead throughput (per district + global per hour).

## Acceptance criteria for this matrix

- Every runtime incident type is represented for both `SUCCEEDED` and `FAILED` outcomes.
- CIVSUB-originated lead permutations are explicitly represented.
- Each permutation has a deterministic integration decision (`YES (Now)`, `YES (Guarded)`, `LATER`, `NO`).
- Each `YES*` decision states exactly how integration should occur.
