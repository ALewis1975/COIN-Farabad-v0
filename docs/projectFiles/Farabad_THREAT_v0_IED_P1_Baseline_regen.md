# Farabad COIN — Threat System v0 + IED Phase 1 (Implementation Baseline)

**Project:** COIN_Farabad (Farabad AO)  
**Timeframe:** 2011-era COIN (Takistan AO)  
**Doc type:** Locked baseline (implementation planning)  
**Version:** 0.1  
**Date:** 2026-01-02  
**Owner:** Mission development (server-authoritative systems)

---

## 0) Purpose

This baseline defines the minimum viable **Threat System** for Farabad:

1. **Threat v0 (Priority 1):** a thin, low-risk **recordkeeping + hook layer** that tracks “threats” as records with lifecycle states, without changing pacing (no global scheduler, no new escalation logic).
2. **IED Phase 1 (Priority 2):** a single, contained “Threat package” type: **`IED_SUSPICIOUS_OBJECT`**, spawned only inside the player bubble (or after AO activation), with full logging and cleanup discipline.
3. **CIVSUB skeleton (Priority 3):** deferred until Threat v0 + IED P1 are stable; implement district scores + identity stubs first (no spawned civilians yet).

This document aligns with Farabad’s cross-cutting standards:
- server is **single writer** for persistent state
- **consumers never guess** (prefer bounded snapshots / explicit transitions)
- log critical transitions
- persistence-safe, cleanup-safe, idempotent behavior

---

## 1) Scope and non-goals

### 1.1 In scope (v0 + P1)
- Threat records exist as **structured state objects** with lifecycle timestamps.
- Threat record **creation** on:
  - **AO activation** (task becomes active/activated), and/or
  - **lead consumption** (optional in v0; same API).
- Threat record **closure** on:
  - incident close / thread on incident closed
- Threat record **observability**:
  - structured OPS logging
  - DEBUG inspector exposure (counts + last event + record dump)
- IED Phase 1:
  - spawn a single “suspicious object” manifestation
  - capture: creation → discovery/interaction → neutralization → cleanup

### 1.2 Explicitly out of scope (v0)
- No district “threat economy” / attack budget
- No global scheduler / autonomous event generator
- No VBIED / suicide bomber logic
- No influence-driven threat probability gates (reserved for CIVSUB coupling later)
- No new tasks/leads generated by Threat (v0 is read-mostly and observational)

## 2) Integration points (minimal hooks, minimal regression)

Threat v0 requires exactly two engine touchpoints:

### 2.1 AO activation hook (create records)
**Location:** `fn_execTickActive.sqf`  
**Existing behavior:** logs `AO_ACTIVATED` and calls `opsPatrolOnActivate`  
**Threat addition:** call a single wrapper that can create threat records for tasks activated by the AO.

**Call (server-only):**
- `["AO_ACTIVATED", _aoContext] call ARC_fnc_threatOnAOActivated;`

Where `_aoContext` is a small, structured object you can assemble with what the AO code already has:
- `ao_id`
- `task_ids_activated[]` (or a best-effort list)
- `pos / marker / radius` (AO center)
- `district_id` (optional, if AO already knows it)

**Hard requirement:** idempotent creation (if called twice, no duplicate records).

### 2.2 Incident close hook (close records)
**Location:** `fn_incidentClose.sqf` and/or `fn_threadOnIncidentClosed.sqf`  
**Threat addition:** close any threat records linked to the incident/task.

**Call (server-only):**
- `["INCIDENT_CLOSED", _incidentContext] call ARC_fnc_threatOnIncidentClosed;`

Where `_incidentContext` includes:
- `incident_id`
- `task_id` (if known)
- `result` (SUCCESS/FAIL/ABORT/etc, as your incident system defines)
- `pos / grid` (best effort)
- `notes` (optional)

**Hard requirement:** safe no-op if no threat record exists.

## 3) Threat v0 data model

### 3.1 IDs and stable keys
Threat IDs should be stable, readable, and sortable.

**Recommended format:**
- `THR:<district_id>:<seq6>`
- Example: `THR:D01:000123`

If no district is known at creation time, use:
- `THR:D00:000123` and optionally patch district later.

### 3.2 Enums (v0 locked set)

```text
THREAT_TYPE:
  IED
  INTIMIDATION
  SMALL_ARMS
  OTHER

THREAT_SUBTYPE:
  # v0 only needs one real subtype (Phase 1)
  IED_SUSPICIOUS_OBJECT

THREAT_STATE:
  CREATED        # record exists, no world manifestation required
  ACTIVE         # associated task/lead is active; may spawn manifestation
  DISCOVERED     # players have discovered/interacted
  NEUTRALIZED    # players neutralized/cleared it (or “resolved” non-IED)
  CLOSED         # incident closed (campaign meaning)
  CLEANED        # world refs removed, record becomes history-only
  EXPIRED        # optional end state if task/lead invalidated

THREAT_AUTHOR:
  SYSTEM
  GM
  DEBUG
```

## 4) Threat v0 storage + persistence keys

Threat v0 should follow the same “versioned store + bounded indexes” pattern used elsewhere in Farabad subsystems.

### 4.1 Runtime keys (missionNamespace, server authoritative)

```text
threat_v0_enabled = true
threat_v0_version = 0
threat_v0_campaign_id = "GUID"

threat_v0_seq = 0
threat_v0_records = HashMap          # threat_id -> ThreatRecord
threat_v0_open_index = Array         # active-ish threats (CREATED/ACTIVE/DISCOVERED/NEUTRALIZED)
threat_v0_closed_index = Array       # bounded history list
threat_v0_closed_max = 200           # cap to prevent unbounded growth

# Debug mirror (optional)
threat_v0_debug_last_event = {}
```

### 4.2 Persistence (profileNamespace or ARC_state sub-blob)

**Preferred:** integrate as a bounded sub-blob inside your existing `ARC_state` persistence object.

**Alternative:** a separate JSON blob in `profileNamespace`:
- `ARC_threat_v0_state_json`

**Hard requirement:** include `threat_v0_version` and support a “reset persistence” workflow.

## 5) Threat v0 API surface (server-side functions)

Keep the public surface small; everything else can be private helpers.

### 5.1 Required functions (v0)

- `ARC_fnc_threatInit`
  - Initializes keys and loads persisted state (if enabled).
- `ARC_fnc_threatCreateFromTask`
  - Idempotently creates a ThreatRecord linked to a task.
- `ARC_fnc_threatUpdateState`
  - Applies state transitions and timestamps, increments `rev`, writes OPS log.
- `ARC_fnc_threatOnAOActivated`
  - Wrapper called by `fn_execTickActive.sqf`; figures out what to create.
- `ARC_fnc_threatOnIncidentClosed`
  - Wrapper called by incident close; closes linked threat(s), logs, schedules cleanup.
- `ARC_fnc_threatDebugSnapshot`
  - Returns counts, open list, last event.

### 5.2 Optional functions (safe to defer)

- `ARC_fnc_threatCreateFromLead`
  - Create ThreatRecord when a lead is consumed/promoted (same schema, link to lead_id).
- `ARC_fnc_threatEmitEvent`
  - Emits a ThreatEvent envelope for UI (delta-bundle style). Not required for v0.

## 6) Logging spec (OPS via ARC_fnc_intelLog)

Threat v0 is mostly an observability feature. Logging is mandatory.

### 6.1 Log category
- Category: `OPS`
- Function: `ARC_fnc_intelLog`

### 6.2 Required log events (v0)
- `THREAT_CREATED`
- `THREAT_STATE_CHANGED`
- `THREAT_CLOSED`
- `THREAT_CLEANED`

### 6.3 Required structured meta (minimum)

```json
{
  "event": "THREAT_STATE_CHANGED",
  "threat_id": "THR:D01:000123",
  "type": "IED",
  "subtype": "IED_SUSPICIOUS_OBJECT",
  "state_from": "ACTIVE",
  "state_to": "DISCOVERED",

  "ao_id": "",
  "district_id": "D01",
  "task_id": "",
  "lead_id": "",
  "incident_id": "",

  "pos": [0,0,0],
  "grid": "",

  "world_spawned": true,
  "world_object_count": 1,

  "note": ""
}
```

### 6.4 Idempotency + “no double logs”
State transitions must be guarded:
- If `state_to == current_state`, do nothing (or log only in DEBUG mode).

## 8) IED Phase 1 — Threat package: IED_SUSPICIOUS_OBJECT

### 8.1 Design intent
Phase 1 is a single contained threat type with tight guardrails:
- spawns only when relevant (AO active and/or player bubble)
- never leaves orphaned objects/groups
- provides fair, readable gameplay cues
- records every step in OPS logs

This is a gameplay abstraction; do not model real-world devices.

### 8.2 Activation rules (v1 locked)
A suspicious object manifestation may be spawned only if all are true:
1. A linked task/lead is **ACTIVE** (ThreatRecord state `ACTIVE`)
2. A player is within the spawn envelope (player bubble), OR the AO has just activated and a player is already present in the AO
3. The threat record is not already `world.spawned=true`

### 8.3 Spawn geometry (use existing mission settings)
Use the existing IED task UX settings (already in `initServer`):
- `ARC_taskMarkerSuspiciousAreaEnabled`
- “IED search radius/jitter” settings

Phase 1 should interpret these as:
- spawn position = task/AO center ± jitter
- objective radius = configured suspicious area radius
- never spawn outside AO bounds (if AO bounds exist)

### 8.4 World manifestation (Phase 1)
- Spawn one object representing a suspicious find.
- Register it immediately with the cleanup system using a stable label:
  - `world.cleanup_label = "THREAT:IED:<threat_id>"`

No complex detonation logic is required for Phase 1. The “threat” is:
- found and interacted with
- then neutralized (defused/removed), OR
- destroyed (if your rules allow), OR
- cleaned up when the incident closes

### 8.5 Required lifecycle + logs (Phase 1)

| Step | Threat state | OPS log event |
|---|---|---|
| record created on AO/task activation | CREATED → ACTIVE | THREAT_CREATED / THREAT_STATE_CHANGED |
| object spawned | ACTIVE (world.spawned=true) | THREAT_STATE_CHANGED (note=SPAWNED) |
| player discovery/interaction | ACTIVE → DISCOVERED | THREAT_STATE_CHANGED |
| neutralization | DISCOVERED → NEUTRALIZED | THREAT_STATE_CHANGED |
| incident closure | * → CLOSED | THREAT_CLOSED |
| cleanup/despawn | * → CLEANED | THREAT_CLEANED |

Phase 1 must also log:
- explicit cleanup/despawn actions (even if neutralized already)

### 8.6 Discovery/interaction detection (keep simple)
Pick one of these approaches (Phase 1 can start with the simplest):
- **ACE interaction action** added to the object:
  - “Inspect suspicious object” → marks DISCOVERED
  - “Neutralize” → marks NEUTRALIZED and deletes object
- **EODS+ integration** (optional if reliable):
  - treat EODS+ “safe” action as NEUTRALIZED
- **Fallback:** object destroyed by damage → mark NEUTRALIZED (or FAILED, if desired), then cleanup

**Hard requirement:** server is the single writer. Client interactions request; server mutates record/state.

### 8.7 Cleanup discipline (mandatory)
- Every spawned object must be registered for cleanup (label above).
- On cleanup/despawn:
  - remove object(s)
  - clear `world.*NetIds`
  - set `world.spawned=false`
  - set state to `CLEANED`
  - write OPS log entry

## 9) Acceptance tests (v0 + IED P1)

### 9.1 Threat v0
- **No pacing changes:** AO activation and incident close behavior unchanged except additional logs.
- **Idempotent creation:** activating the same AO twice does not duplicate threats.
- **Safe no-op close:** closing an incident with no linked threat does not error.
- **Bounded history:** closed index does not grow unbounded.

### 9.2 IED Phase 1
- **Bubble gating:** object never spawns outside player relevance bubble (unless AO activation already has players present).
- **Cleanup safe:** objects are removed on incident close or when players leave and cleanup triggers.
- **Logging complete:** creation / discovery / neutralization / cleanup all appear in OPS logs with threat_id and location.
- **Regression guard:** restarting mission with persistence enabled does not duplicate spawns.

## 10) Priority 3 — CIVSUB skeleton (deferred until v0 + P1 stable)

CIVSUB should not begin until Threat v0 + IED P1 are stable, so failures are attributable (state vs cleanup vs subsystem).

When started, do it in this order:
1. **District metadata + decay tick only** (no physical civilians)
2. **Delta bundle emission** (district deltas + touched identity updates)
3. **Only then** add physical civilian sampling + ACE interactions

State storage should be either:
- separate JSON blob in `profileNamespace`, or
- a bounded `ARC_state` sub-blob with explicit versioning

(See CIVSUB baseline for exact district schema and identity model.)

---

## 11) Roadmap (post-v0)

After Threat v0 + IED Phase 1:
- Add a minimal per-district gating layer that consumes CIVSUB scores (probability gates).
- Expand threat types (VBIED/Suicide) using the existing IED/VBIED/Suicide planning spec.
- Consider adding ThreatEvent delta bundles if/when UI consumes threat activity as a live feed.
